# 关键决策

## 决策 1：采用纯静态架构（HTML/CSS/JS）

- 决策：不引入前端框架与打包工具，直接静态文件开发。
- 原因：项目功能聚焦，用户以快速上线与易维护为目标。
- 结果：部署简单，Vercel 配置仅需 `Other + 无构建`。

## 决策 2：阶段声音采用“呼气采样 + 其余合成”

- 决策：呼气用真实采样，吸气/暂停保留合成提示音。
- 原因：呼气对听感最敏感，采样更自然；其余合成可控且轻量。
- 结果：兼顾听感与实现复杂度。

## 决策 3：呼气采样格式切换到 MP3

- 决策：将呼气主采样从 `OGG` 切换到 `MP3`。
- 原因：移动端浏览器（尤其部分国产内核）对 OGG 兼容不稳定。
- 结果：手机 Chrome 与主流浏览器表现更稳定，线上一致性更好。
- 对应提交：`939e951`。

## 决策 4：导出失败时提供兜底能力

- 决策：在受浏览器限制的场景（常见于 `file://`）导出不直接失败，尝试兜底继续完成。
- 原因：保证“可以导出”优先，减少用户中断感。
- 结果：`file://` 下成功率提升，但最佳实践仍是本地 HTTP/线上访问。

## 决策 5：代码托管 + 自动部署

- 决策：GitHub 托管、Vercel 自动部署。
- 原因：方便版本追踪、回滚、分享和后续迭代。
- 结果：只要推送 `main` 分支，Vercel 自动更新生产站点。

## 决策 6：文档集中到 `docs/`

- 决策：需求、技术方案与迭代记录统一放在 `docs/`。
- 原因：便于后续让 AI 或协作者快速接手上下文。
- 结果：维护成本降低，迭代信息可追溯。
